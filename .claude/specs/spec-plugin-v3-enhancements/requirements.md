# Requirements: spec-plugin-v3-enhancements

> Generated by spec-driven plugin (spec-planner agent, Opus 4.6)

## Overview

Enhance the spec-driven development plugin from v2.2.0 to v3.0.0. This release introduces git worktree isolation for safe parallel execution, crash recovery via checkpoint commits, a missing spec-retro.sh script, cross-spec dependency tracking, spec presets for common patterns, a `/spec-import` command for converting existing PRDs to EARS notation, and an improved init.sh template. The version bump to 3.0.0 reflects the new default worktree behavior (a behavioral change to all execution scripts).

(none)

## User Stories

### US-1: Git Worktree Isolation

**As a** plugin user (developer)
**I want** spec execution scripts to run in an isolated git worktree by default
**So that** in-progress spec work does not pollute my main branch and failed executions can be discarded without affecting the working tree

#### Acceptance Criteria (EARS Notation)

1. WHEN spec-exec.sh, spec-loop.sh, or spec-team.sh is invoked without the `--no-worktree` flag
   THE SYSTEM SHALL create a git worktree at `.claude/specs/.worktrees/<spec-name>/` on branch `spec/<spec-name>` before executing Claude

2. WHEN a worktree already exists at `.claude/specs/.worktrees/<spec-name>/`
   THE SYSTEM SHALL reuse the existing worktree without error and ensure the `spec/<spec-name>` branch is checked out

3. WHEN the branch `spec/<spec-name>` already exists in the repository
   THE SYSTEM SHALL check out the existing branch in the worktree without attempting to create a duplicate branch

4. WHEN the `--no-worktree` flag is passed to any execution script
   THE SYSTEM SHALL execute directly in the current working directory without creating a worktree, preserving v2.x behavior

5. WHEN all tasks are verified and the execution script outputs `<promise>COMPLETE</promise>`
   THE SYSTEM SHALL print a suggested `gh pr create --head spec/<spec-name> --title "<spec-name>"` command to stdout

6. WHEN execution fails or the process is interrupted (non-zero exit, SIGINT, SIGTERM)
   THE SYSTEM SHALL leave the worktree in place so the user can inspect or remove it manually

7. THE SYSTEM SHALL ensure `.claude/specs/.worktrees/` is listed in the project root `.gitignore`

8. WHEN Claude runs inside a worktree
   THE SYSTEM SHALL set the working directory to the worktree path for the `claude` CLI invocation

9. WHEN the script is run in a directory that is not a git repository
   THE SYSTEM SHALL print an error and exit with code 1 (existing behavior, no change)

### US-2: Crash Recovery via Git Checkpoint Commits

**As a** plugin user (developer)
**I want** each loop iteration to create a checkpoint commit before running Claude
**So that** if Claude crashes or exits non-zero, all code and spec state can be cleanly rolled back to a known-good point

#### Acceptance Criteria (EARS Notation)

1. WHEN a new iteration begins in spec-loop.sh or spec-team.sh
   THE SYSTEM SHALL stage all changes and create a temporary checkpoint commit with message `checkpoint: pre-iteration <N>` before invoking Claude

2. WHEN there are no uncommitted changes (staged or unstaged) at the start of an iteration
   THE SYSTEM SHALL skip the checkpoint commit for that iteration

3. WHEN Claude exits with a non-zero exit code and a checkpoint commit was created for the current iteration
   THE SYSTEM SHALL execute `git reset --hard <checkpoint-sha>` and print "Rolled back to checkpoint: pre-iteration <N>" to stderr

4. WHEN Claude exits with exit code zero and a checkpoint commit was created
   THE SYSTEM SHALL NOT roll back; Claude's own commit(s) supersede the checkpoint

5. WHEN Claude exits with exit code zero but does not create its own commit, and a checkpoint was created
   THE SYSTEM SHALL leave the checkpoint commit in place (it contains the valid pre-iteration state)

6. THE SYSTEM SHALL NOT implement checkpoint recovery in spec-exec.sh (single-shot execution has no loop to recover within)

7. WHEN `--no-worktree` is used, checkpoint commits occur on the current branch
   THE SYSTEM SHALL still create and manage checkpoint commits in the same manner

### US-3: spec-retro.sh Script

**As a** plugin user (developer)
**I want** a non-interactive `spec-retro.sh` script following the same pattern as other post-implementation scripts
**So that** I can generate retrospectives in CI/CD or headless environments

#### Acceptance Criteria (EARS Notation)

1. WHEN spec-retro.sh is invoked with `--spec-name <name>`
   THE SYSTEM SHALL locate the spec at `.claude/specs/<name>/` and read requirements.md, design.md, tasks.md, and progress.md

2. WHEN spec-retro.sh is invoked without `--spec-name` and exactly one spec exists in `.claude/specs/`
   THE SYSTEM SHALL auto-detect the spec name

3. WHEN spec-retro.sh is invoked without `--spec-name` and multiple specs exist
   THE SYSTEM SHALL print an error listing available specs and exit with code 1

4. WHEN spec-retro.sh builds the Claude prompt
   THE SYSTEM SHALL include instructions to analyze: iteration count from progress.md, debugging cycles (tasks with multiple status changes), commit frequency and fix-vs-feature ratio from git log, time span, what went well, what caused friction, root causes, and action items

5. WHEN spec-retro.sh completes successfully
   THE SYSTEM SHALL have written `retro.md` to the spec directory and output `<promise>RETRO_COMPLETE</promise>` to stdout

6. WHEN progress.md does not exist for the spec
   THE SYSTEM SHALL proceed with available data and instruct Claude to note the missing progress log in the retrospective

7. THE SYSTEM SHALL invoke `claude --dangerously-skip-permissions -p "<prompt>"` for non-interactive execution, matching the invocation pattern of spec-accept.sh

8. THE SYSTEM SHALL accept `--spec-name` as its only argument, following the minimal argument pattern of spec-accept.sh

### US-4: init.sh Template Enhancement

**As a** plugin user (developer)
**I want** the init.sh template to include comprehensive sections with multi-stack examples
**So that** I have a clear starting point for configuring my project regardless of tech stack

#### Acceptance Criteria (EARS Notation)

1. WHEN `/spec <name>` copies templates to a new spec directory
   THE SYSTEM SHALL include the enhanced init.sh with sections for: dependency installation, environment variable setup, dev server start, health check, and test runner

2. THE SYSTEM SHALL include commented-out example commands for at least three common stacks (Node.js/npm, Python/pip, Go) in each section

3. THE SYSTEM SHALL include a header comment explaining that execution scripts (spec-exec.sh, spec-loop.sh, spec-team.sh) read this file to understand how to run the project

4. WHEN init.sh has not been customized by the user
   THE SYSTEM SHALL NOT execute any commands, as all actionable lines are commented out by default

5. THE SYSTEM SHALL update the existing `templates/init.sh` file in place (not create a separate file)

### US-5: Cross-Spec Dependencies

**As a** plugin user (developer)
**I want** to declare that one spec depends on another spec being complete
**So that** execution scripts enforce ordering and I can see cross-spec status at a glance

#### Acceptance Criteria (EARS Notation)

1. WHEN a requirements.md contains a `## Depends On` section with a bulleted list of spec names
   THE SYSTEM SHALL parse each bullet as a dependency spec name

2. WHEN spec-exec.sh, spec-loop.sh, or spec-team.sh begins execution for a spec with dependencies
   THE SYSTEM SHALL check each dependency spec's tasks.md to determine if ALL tasks have Status: completed AND Wired: yes or n/a AND Verified: yes

3. WHEN any dependency spec has incomplete tasks
   THE SYSTEM SHALL print an error listing each incomplete dependency with its name and percentage of verified tasks, then exit with code 1

4. WHEN a dependency references a spec name whose directory does not exist in `.claude/specs/`
   THE SYSTEM SHALL print "Error: Dependency spec not found: <name>" and exit with code 1

5. WHEN `/spec-status` is run for a spec that has dependencies
   THE SYSTEM SHALL include a `### Dependencies` section showing each dependency name, its status (completed or X/Y tasks verified), and whether it blocks execution

6. WHEN two or more specs form a circular dependency (A depends on B, B depends on A, or longer cycles)
   THE SYSTEM SHALL detect the cycle during dependency checking, print "Error: Circular dependency detected: A -> B -> A", and exit with code 1

7. WHEN a spec has no `## Depends On` section in requirements.md
   THE SYSTEM SHALL treat it as having zero dependencies (preserving current behavior)

8. THE SYSTEM SHALL add a `## Depends On` section with a placeholder comment to the `templates/requirements.md` template

### US-6: Spec Templates (Presets)

**As a** plugin user (developer)
**I want** preset spec templates for common development patterns
**So that** I can bootstrap a spec with relevant user stories and acceptance criteria instead of starting blank

#### Acceptance Criteria (EARS Notation)

1. WHEN `/spec <name>` begins the interactive requirements gathering phase
   THE SYSTEM SHALL ask the user via AskUserQuestion whether to start from a preset or from scratch, presenting available presets as options

2. WHEN the user selects the "REST API" preset
   THE SYSTEM SHALL pre-fill requirements.md with user stories for: CRUD operations, input validation, authentication/authorization, error responses, and pagination

3. WHEN the user selects the "React Page" preset
   THE SYSTEM SHALL pre-fill requirements.md with user stories for: component rendering, routing, state management, API integration, loading/error states, and responsive layout

4. WHEN the user selects the "CLI Tool" preset
   THE SYSTEM SHALL pre-fill requirements.md with user stories for: argument parsing, subcommands, output formatting (table, JSON, plain), error handling, and help/version text

5. WHEN the user selects "Start from scratch"
   THE SYSTEM SHALL use the existing blank requirements.md template, preserving current behavior

6. THE SYSTEM SHALL store preset files in `templates/presets/` as `rest-api.md`, `react-page.md`, and `cli-tool.md`

7. WHEN a preset is selected, the spec-planner agent
   THE SYSTEM SHALL use the preset as a starting point and customize it based on the user's specific answers (not use it verbatim)

### US-7: Spec Import from Markdown/PRD

**As a** plugin user (developer)
**I want** to import an existing markdown document and have it converted into a spec with EARS notation requirements
**So that** I can reuse existing PRDs, RFCs, and design docs without manually rewriting them

#### Acceptance Criteria (EARS Notation)

1. WHEN `/spec-import <feature-name> --file <path>` is invoked with a path to an existing markdown file
   THE SYSTEM SHALL read the file, create `.claude/specs/<feature-name>/`, and generate requirements.md with user stories and EARS acceptance criteria derived from the document

2. WHEN the import processes the source document
   THE SYSTEM SHALL delegate to Claude (via Task tool to the spec-planner agent) with instructions to extract requirements, user roles, acceptance criteria, and constraints from the document content

3. WHEN the source document contains no discernible requirements, user stories, or feature descriptions
   THE SYSTEM SHALL generate a best-effort requirements.md from whatever goals or descriptions are present and print "Warning: Source document had limited requirements content. Manual review is strongly recommended."

4. WHEN the file path does not exist
   THE SYSTEM SHALL print "Error: File not found: <path>" and exit without creating a spec directory

5. WHEN the file does not have a .md or .markdown extension
   THE SYSTEM SHALL print "Warning: File may not be markdown. Proceeding with import." and continue

6. WHEN the import generates requirements.md
   THE SYSTEM SHALL NOT generate design.md or tasks.md; the user proceeds to design via `/spec-refine` or the normal `/spec` workflow

7. THE SYSTEM SHALL create `commands/spec-import.md` as a new slash command with allowed-tools including Read, Write, Glob, Grep, Task, and AskUserQuestion

8. WHEN the import completes successfully
   THE SYSTEM SHALL print "Import complete. Review requirements at .claude/specs/<feature-name>/requirements.md" and suggest running `/spec-refine` to adjust before proceeding to design

## Non-Functional Requirements

### NFR-1: Shell Compatibility
- All new and modified shell scripts MUST use the same bash constructs as existing scripts (e.g., `[[ ]]`, arrays, `$(command)` substitution)
- Scripts MUST work on macOS (Darwin, default zsh with bash available) and Linux (bash 4+)
- No new external tool dependencies beyond: git (2.5+ for worktrees), gh (for PR suggestion), claude CLI

### NFR-2: Performance
- Git worktree creation MUST complete in under 2 seconds for repositories with up to 10,000 tracked files
- Cross-spec dependency checking MUST complete in under 1 second when checking up to 20 dependency specs
- Checkpoint commit creation MUST complete in under 1 second for typical working trees

### NFR-3: Reliability
- Crash recovery MUST restore the branch to a clean state after any non-zero exit from Claude
- Worktree creation MUST be idempotent: running the same script twice with the same spec name reuses the existing worktree
- Dependency checking MUST be deterministic: the same spec state always produces the same pass/fail outcome

### NFR-4: Backward Compatibility
- All v2.x invocations of execution scripts (without new flags) MUST continue to work; the only behavioral change is the new default worktree isolation
- `--no-worktree` provides explicit opt-out from worktree mode
- Specs without `## Depends On` MUST be treated as having no dependencies
- `/spec` without selecting a preset MUST behave identically to v2.x

### NFR-5: Minimal Engineering
- No new runtime dependencies (no npm, pip, or external packages)
- Presets stored as plain markdown files in `templates/presets/`
- Import parsing performed entirely through Claude CLI prompt (no separate NLP or parsing library)

## Out of Scope

- Metrics dashboard or visualization for iteration timing and task metrics
- Task dependency graph visualization (visual DAG rendering)
- Multi-spec pipeline orchestrator (running multiple specs in sequence/parallel)
- Import from external tools (Linear, Jira, Notion) -- only local markdown files
- Automatic worktree cleanup on script failure (user manages via `git worktree remove`)
- Automatic PR creation (only the `gh pr create` command is printed for the user to copy/run)
- Changes to agent model routing (Opus/Sonnet assignments remain as-is)
- Changes to spec-team.sh agent team composition (Implementer, Tester, Reviewer, Debugger)

## Assumptions

1. `git` CLI is available in PATH and the project is a git repository (already required by existing scripts)
2. `gh` CLI may or may not be installed; the PR command is only printed as a suggestion, not executed
3. `claude` CLI supports `--dangerously-skip-permissions -p` (already used by all existing scripts)
4. Git 2.5+ is available (for `git worktree` support; released July 2015)
5. The `[[ ]]` syntax used throughout existing scripts is the accepted baseline for this project
6. Checkpoint commits on the worktree branch are acceptable workflow artifacts; they get squashed or cleaned during PR review
7. The existing `templates/init.sh` is enhanced in-place (not replaced with a new file at a different path)
8. Cross-spec dependency metadata lives in requirements.md to keep spec artifacts self-contained
9. `/spec-import` uses Claude for parsing because it must work offline without external NLP services
10. The `.claude/specs/.worktrees/` directory is created at the project root, adjacent to the existing `.claude/specs/` structure
