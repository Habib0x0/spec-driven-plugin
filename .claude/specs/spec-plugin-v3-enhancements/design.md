# Design: spec-plugin-v3-enhancements

> Generated by spec-driven plugin (spec-planner agent, Opus 4.6)

## Overview

This design covers eight enhancements to the spec-driven plugin, upgrading it from v2.2.0 to v3.0.0. The changes primarily affect execution scripts (spec-exec.sh, spec-loop.sh, spec-team.sh), the `/spec` command workflow, templates, and introduce one new command (`/spec-import`) and one new script (`spec-retro.sh`). The architecture preserves the existing pattern of "bash script builds prompt, invokes claude CLI" while adding git worktree management and cross-spec dependency checking as pre-execution steps.

### Data Flow

#### Execution with Worktree (US-1 + US-2)

```
User invokes spec-loop.sh --spec-name my-feature
    │
    ├── 1. Parse args, auto-detect spec
    ├── 2. Check cross-spec dependencies (lib/deps.sh)
    │      ├── Parse requirements.md "## Depends On"
    │      ├── For each dep: check .claude/specs/<dep>/tasks.md
    │      └── If any incomplete → error + exit 1
    ├── 3. Setup worktree (lib/worktree.sh)
    │      ├── mkdir -p .claude/specs/.worktrees/<spec-name>/
    │      ├── git worktree add (or reuse existing)
    │      └── Set WORK_DIR to worktree path
    ├── 4. Loop iterations (in WORK_DIR):
    │      ├── 4a. Create checkpoint commit (lib/checkpoint.sh)
    │      │       ├── git add -A && git commit -m "checkpoint: pre-iteration N"
    │      │       └── Store CHECKPOINT_SHA
    │      ├── 4b. Build prompt, invoke claude
    │      ├── 4c. Check exit code
    │      │       ├── Non-zero → git reset --hard $CHECKPOINT_SHA
    │      │       └── Zero → continue
    │      └── 4d. Check for COMPLETE marker
    ├── 5. On COMPLETE:
    │      └── Print: gh pr create --head spec/my-feature --title "my-feature"
    └── Done
```

#### Spec Import Flow (US-7)

```
User invokes /spec-import my-feature --file /path/to/prd.md
    │
    ├── 1. Validate file exists and is readable
    ├── 2. Read file content
    ├── 3. Create .claude/specs/my-feature/ directory
    ├── 4. Delegate to spec-planner agent via Task tool
    │      ├── Pass file content as context
    │      ├── Instruct: extract requirements, write EARS notation
    │      └── Agent writes requirements.md
    ├── 5. Print import summary and suggestion to /spec-refine
    └── Done
```

## Components

### C-1: Worktree Manager (lib/worktree.sh)

**Purpose**: Encapsulate all git worktree creation, reuse, and configuration logic into a sourceable shell library.

**Responsibilities**:
- Create worktree at `.claude/specs/.worktrees/<spec-name>/` if it does not exist
- Reuse existing worktree if it already exists at that path
- Create branch `spec/<spec-name>` if it does not exist, or check it out if it does
- Export `WORK_DIR` variable pointing to the worktree path (or `pwd` if `--no-worktree`)
- Ensure `.gitignore` includes `.claude/specs/.worktrees/`
- Print the `gh pr create` suggestion on completion

**Interface**:
```bash
# Source the library
source "$SCRIPT_DIR/lib/worktree.sh"

# Setup worktree. Sets WORK_DIR.
# Args: spec_name, use_worktree (true/false)
setup_worktree "$SPEC_NAME" "$USE_WORKTREE"

# Print PR suggestion after completion
print_pr_suggestion "$SPEC_NAME"
```

**Error Handling**:
- If `git worktree add` fails (e.g., branch checked out elsewhere), print error and exit 1
- If the path exists but is not a valid worktree (corrupted), remove it and recreate

**State**:
- `WORK_DIR` exported environment variable
- `USE_WORKTREE` flag (true/false)

---

### C-2: Checkpoint Manager (lib/checkpoint.sh)

**Purpose**: Encapsulate checkpoint commit creation and rollback logic into a sourceable shell library.

**Responsibilities**:
- Create a checkpoint commit with all current changes before each iteration
- Skip checkpoint if there are no uncommitted changes
- Roll back to checkpoint on Claude failure (non-zero exit)
- Track checkpoint SHA for the current iteration

**Interface**:
```bash
# Source the library
source "$SCRIPT_DIR/lib/checkpoint.sh"

# Create checkpoint before iteration. Sets CHECKPOINT_SHA (empty if skipped).
# Args: iteration_number, work_dir
create_checkpoint "$ITERATION" "$WORK_DIR"

# Handle post-iteration recovery.
# Args: exit_code, checkpoint_sha, iteration_number, work_dir
handle_checkpoint_recovery "$EXIT_CODE" "$CHECKPOINT_SHA" "$ITERATION" "$WORK_DIR"
```

**Error Handling**:
- If `git commit` fails during checkpoint (unusual, since we check for changes first), log warning and continue without checkpoint protection
- If `git reset --hard` fails during rollback, print critical error directing user to manually inspect the branch

**State**:
- `CHECKPOINT_SHA` variable per iteration (empty string means no checkpoint)

---

### C-3: Dependency Checker (lib/deps.sh)

**Purpose**: Parse cross-spec dependencies from requirements.md and verify all dependencies are complete before execution.

**Responsibilities**:
- Parse `## Depends On` section from requirements.md
- For each dependency, read its tasks.md and check completion status
- Detect circular dependencies via depth-first traversal
- Report blocking dependencies with names and completion percentages
- Provide dependency status data for `/spec-status`

**Interface**:
```bash
# Source the library
source "$SCRIPT_DIR/lib/deps.sh"

# Check all dependencies are met. Exits 1 if not.
# Args: spec_name
check_dependencies "$SPEC_NAME"

# Get dependency status for display (does not exit on failure).
# Args: spec_name
# Outputs: lines of "dep_name:status:verified_count:total_count"
get_dependency_status "$SPEC_NAME"
```

**Algorithm for Circular Dependency Detection**:
```
visited = {}
path = {}

function check_cycle(spec_name):
    if spec_name in path:
        print cycle from path[spec_name] to spec_name
        exit 1
    if spec_name in visited:
        return  # already checked, no cycle
    path[spec_name] = true
    visited[spec_name] = true
    for each dep in parse_depends_on(spec_name):
        check_cycle(dep)
    remove spec_name from path
```

Since bash does not have native sets/maps, the implementation uses temporary files or associative arrays (bash 4+ on Linux, available on macOS with Homebrew bash). Given that existing scripts already use `[[ ]]` and arrays, associative arrays for cycle detection with a fallback to file-based tracking is the pragmatic approach. For portability, the design uses file-based tracking (one temp file per visited node).

**Error Handling**:
- Missing dependency directory: print "Error: Dependency spec not found: <name>" and exit 1
- Missing tasks.md in dependency: print "Error: Dependency <name> has no tasks.md" and exit 1
- Circular dependency: print the cycle chain and exit 1

### C-4: spec-retro.sh Script

**Purpose**: Non-interactive retrospective generation following the pattern of existing post-implementation scripts.

**Responsibilities**:
- Parse arguments and auto-detect spec name
- Validate spec directory and required files exist
- Build a Claude prompt with spec artifacts and git history
- Invoke `claude --dangerously-skip-permissions -p` with the prompt
- Output `<promise>RETRO_COMPLETE</promise>` on success

**Interface**:
```bash
spec-retro.sh [--spec-name <name>]
```

**Prompt Structure** (mirrors spec-accept.sh pattern):
1. Header: "# Retrospective Analysis: <spec-name>"
2. Spec files: requirements.md, design.md, tasks.md, progress.md
3. Instructions for Claude:
   - Analyze iteration count from progress.md
   - Analyze debugging cycles (tasks attempted multiple times)
   - Check git log for commit patterns
   - Identify what went well and what caused friction
   - Generate root cause analysis
   - Write action items
4. Output format: retro.md structure
5. Completion marker: `<promise>RETRO_COMPLETE</promise>`

## Data Models

### Worktree Path Convention

```
.claude/specs/.worktrees/<spec-name>/
```

- The `.worktrees/` directory sits inside `.claude/specs/` to keep all spec artifacts together
- Each spec gets its own subdirectory named exactly after the spec
- Branch name convention: `spec/<spec-name>`

### Dependency Declaration Format (in requirements.md)

```markdown
## Depends On

- auth-system
- database-migrations
```

Parsing rules:
- Section header must be exactly `## Depends On` (case-sensitive)
- Each dependency is a markdown bullet (`- `) followed by the spec name
- Spec name must match a directory name in `.claude/specs/`
- Empty section or no section means no dependencies
- Lines that are not bullets are ignored (allows comments)

### Spec Completion Definition

A spec is "complete" when ALL of the following are true for every task in tasks.md:
- `Status: completed`
- `Wired: yes` OR `Wired: n/a`
- `Verified: yes`

This definition is used by:
1. Cross-spec dependency checking (lib/deps.sh)
2. The COMPLETE marker output in execution scripts
3. `/spec-status` completion percentage

### Checkpoint Commit Convention

```
checkpoint: pre-iteration <N>
```

- Commit message is machine-readable for potential future tooling
- Created on the current branch (typically `spec/<spec-name>` in worktree mode)
- Contains all staged and unstaged changes at iteration start

## Implementation Considerations

### File Organization: Shared Shell Libraries

The worktree, checkpoint, and dependency logic is shared across three scripts (spec-exec.sh, spec-loop.sh, spec-team.sh). Rather than duplicating code, extract shared functions into `scripts/lib/`:

```bash
# At the top of spec-loop.sh:
SCRIPT_DIR="$(cd "$(dirname "$0")" && pwd)"
source "$SCRIPT_DIR/lib/worktree.sh"
source "$SCRIPT_DIR/lib/checkpoint.sh"
source "$SCRIPT_DIR/lib/deps.sh"
```

This follows minimal engineering principles -- three small focused files instead of one monolithic utility script.

### Worktree Integration Points

Each execution script needs these modifications:

1. **Argument parsing**: Add `--no-worktree` flag
2. **Pre-execution**: Source lib/worktree.sh, call `setup_worktree`
3. **Claude invocation**: Change working directory to `$WORK_DIR`
4. **Post-completion**: Call `print_pr_suggestion` when COMPLETE detected

For spec-exec.sh specifically:
- Uses `cd "$WORK_DIR"` before building prompt and invoking Claude
- No checkpoint logic (single iteration)

For spec-loop.sh and spec-team.sh:
- Uses `cd "$WORK_DIR"` at start
- Adds checkpoint create/recover around each iteration
- Spec file paths become relative to WORK_DIR

### Checkpoint Interaction with Worktree

When worktree mode is active:
- Checkpoint commits happen on the `spec/<spec-name>` branch inside the worktree
- This means the main branch is never touched by checkpoints
- Rollbacks (`git reset --hard`) only affect the worktree branch

When `--no-worktree` is used:
- Checkpoint commits happen on whatever branch the user is on
- This is riskier but the user opted in by using `--no-worktree`
- The checkpoint still protects against mid-iteration corruption

### Dependency Checking Placement

Dependency checking runs BEFORE worktree setup:
1. It reads from `.claude/specs/` in the original working directory
2. If deps fail, we exit before any worktree creation (fast fail)
3. This avoids creating a worktree that can't be used

### Claude Working Directory in Worktree Mode

When running in a worktree, the spec files (requirements.md, tasks.md, etc.) are at the same relative path as in the main tree because git worktrees share the same file structure. The key difference:

- `SPEC_DIR` remains `.claude/specs/$SPEC_NAME` (relative)
- `WORK_DIR` changes from `$(pwd)` to `.claude/specs/.worktrees/<spec-name>/`
- Claude is invoked from within `WORK_DIR` so all relative paths work

Important: The worktree contains the full repository, not just the spec directory. This means Claude can read and modify source code, which is the whole point.

### Gitignore Management

The `setup_worktree` function adds `.claude/specs/.worktrees/` to `.gitignore` if not already present. This is done with:

```bash
if ! grep -qF '.claude/specs/.worktrees/' .gitignore 2>/dev/null; then
    echo '.claude/specs/.worktrees/' >> .gitignore
fi
```

This runs on the main working directory (before entering the worktree) and only adds the entry once.

### Exit Code Handling for Checkpoints

The current scripts use `set -e` which causes the script to exit on any non-zero exit code. For checkpoint recovery, we need to capture Claude's exit code without triggering `set -e`. The pattern:

```bash
set +e
claude --dangerously-skip-permissions -p "..." | tee "$OUTPUT_FILE"
CLAUDE_EXIT=${PIPESTATUS[0]}
set -e

handle_checkpoint_recovery "$CLAUDE_EXIT" "$CHECKPOINT_SHA" "$ITERATION" "$WORK_DIR"
```

This temporarily disables errexit to capture the exit code, then re-enables it.

### Template Modification Strategy

**requirements.md template**: Add `## Depends On` section after the overview:
```markdown
## Depends On

<!-- List spec names this spec depends on. Remove this section if none. -->
<!-- - other-spec-name -->
```

**init.sh template**: Replace the current minimal template with the enhanced version containing multi-stack examples.

### Preset Integration with /spec Command

The `/spec` command's interactive flow changes:

```
1. Initialize directory
2. NEW: Ask "Start from a preset?" via AskUserQuestion
   - If preset: read templates/presets/<choice>.md, include in context
   - If scratch: continue normally
3. Interactive requirements gathering (existing)
4. Delegate to spec-planner (existing, but now receives preset context if selected)
5. Tasks phase (existing)
6. Summary (existing)
```

The preset content is passed to the spec-planner agent as "starting context" alongside user answers. The agent is instructed to customize and extend the preset, not use it verbatim.

### Dependencies

- git 2.5+ (for `git worktree` commands)
- gh (GitHub CLI, optional, only for PR suggestion)
- claude CLI (existing dependency)
- bash (existing dependency)

### Security Considerations

- **Worktree isolation**: Worktrees share the `.git` directory with the main repo. This means they have access to the same git hooks, credentials, and configuration. No additional security boundary is created -- worktrees are for branch isolation, not security isolation.
- **Checkpoint commits**: Checkpoint commits may contain sensitive data that was staged. Since they live on a feature branch (not main), exposure is limited. If the branch is pushed, checkpoint commits would be visible. Users should squash before merging.
- **spec-import file reading**: The import command reads a user-specified file path. It passes the content to Claude. No file path validation beyond existence is needed since this runs locally as the user.
- **Script injection**: All scripts use `claude --dangerously-skip-permissions -p "$(cat $PROMPT_FILE)"` which reads from a temp file. The prompt content comes from spec files that the user controls. No additional injection risk beyond the existing design.

### Performance Considerations

- **Worktree creation**: `git worktree add` is fast (creates symlinks to .git, checks out branch). For large repos, the checkout step is the bottleneck. The 2-second target is achievable for repos up to 10k files.
- **Dependency checking**: Reads tasks.md files and counts statuses. Pure text parsing with grep/awk. Trivially fast even for 20+ specs.
- **Checkpoint commits**: `git add -A && git commit` is fast for typical working trees. Large binary files could slow it down, but that is a user concern, not a plugin concern.

### Testing Strategy

- **Shell scripts**: Manual testing with a test git repository. Create specs, run scripts, verify worktree creation, checkpoint behavior, dependency checking.
- **Presets**: Verify each preset produces valid requirements.md structure by running through `/spec` with each preset selected.
- **Import**: Test with various markdown formats (minimal, verbose, non-standard) to verify best-effort parsing.
- **Dependency cycles**: Create A->B->A dependency chain, verify cycle detection works.
- **Edge cases**: Test with existing worktree, existing branch, no changes for checkpoint, missing dependency directory.

## Alternatives Considered

### A-1: Git Branch Instead of Worktree

**Description**: Create a feature branch and switch to it (git checkout) instead of using git worktrees.

**Pros**:
- Simpler implementation (no worktree management)
- Works with older git versions

**Cons**:
- Switches the user's working directory branch, affecting any open editors or file watchers
- Cannot run multiple specs simultaneously (each would need its own branch checkout)
- Interrupted execution leaves the user on the wrong branch

**Decision**: Not chosen. Git worktrees provide true isolation without disrupting the user's main working tree. The git 2.5+ requirement is acceptable (released 2015).

### A-2: Dependency Config in a Separate File

**Description**: Store cross-spec dependencies in a dedicated `.claude/specs/dependencies.json` or similar config file instead of in each requirements.md.

**Pros**:
- Centralized dependency graph
- Easier to parse programmatically (JSON vs markdown)
- No risk of parser confusion with markdown content

**Cons**:
- Adds a new config file format to learn
- Decouples dependency info from the spec it belongs to
- Requires keeping the external file in sync with spec creation/deletion
- Violates the principle that each spec directory is self-contained

**Decision**: Not chosen. Storing dependencies in requirements.md keeps specs self-contained. The parsing is simple (grep for `## Depends On`, then extract bullet items).

### A-3: Checkpoint via Git Stash Instead of Commit

**Description**: Use `git stash` before each iteration and `git stash pop` on failure, instead of checkpoint commits.

**Pros**:
- No checkpoint commits polluting the branch history
- Stash is designed for temporary state saving

**Cons**:
- `git stash pop` can fail with merge conflicts if Claude changed the same files
- Stash does not capture the full commit state (only uncommitted changes)
- If Claude makes commits, there is no clean rollback point
- Stash stack can get confusing if multiple iterations stash

**Decision**: Not chosen. Checkpoint commits are more robust because `git reset --hard` to a known SHA is deterministic and cannot have merge conflicts.

### A-4: Single Utility Script Instead of lib/ Directory

**Description**: Put all shared functions (worktree, checkpoint, deps) in a single `scripts/utils.sh` file.

**Pros**:
- One file to source
- Simpler to discover

**Cons**:
- Violates single-responsibility principle
- Growing file becomes hard to maintain
- spec-exec.sh would source functions it does not need (checkpoint, deps)

**Decision**: Not chosen. Three focused library files in `scripts/lib/` is cleaner and each script only sources what it needs.

### A-5: Import via Dedicated Parser Script

**Description**: Write a bash/python script to parse markdown PRDs into EARS format, rather than using Claude.

**Pros**:
- Deterministic output
- No Claude CLI dependency for import
- Faster execution

**Cons**:
- Requires building/maintaining a markdown-to-EARS parser
- Cannot handle varied document formats without NLP
- Adds complexity that Claude handles naturally
- Would require a new external dependency (python) for non-trivial parsing

**Decision**: Not chosen. Claude excels at understanding varied document formats and producing structured output. Using the Task tool to delegate to the spec-planner agent keeps the implementation minimal and leverages existing infrastructure.
