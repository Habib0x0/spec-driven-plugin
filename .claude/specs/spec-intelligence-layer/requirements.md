# Requirements: Spec Intelligence Layer

> Generated by spec-driven plugin

## Overview

The spec-driven plugin currently suffers from five critical problems: (1) agents write code but do not wire it into the application because they lack project-specific knowledge of registration points, (2) specs implement features in isolation without understanding what a complete application needs, (3) every spec run re-discovers the same codebase patterns from scratch with no persistent memory, (4) spec-loop executes tasks sequentially even when many are independent, and (5) there is no structured debugging workflow within the spec context. This spec introduces seven components to address these problems: Phase 0 Discovery, Project Profile, Wiring-Aware Implementation, Verification Gates, /spec-debug command, Dependency-Aware Parallel Execution, and Regression Awareness.

## User Roles

- **Plugin User**: A developer using the spec-driven plugin to build applications. Runs /spec, /spec-loop, /spec-debug commands and expects generated code to be fully wired and functional.
- **Plugin Developer**: A developer maintaining or extending the spec-driven plugin itself. Modifies agents, commands, scripts, and templates.

## US-1: Phase 0 Discovery Scan

**As a** plugin user
**I want** the plugin to automatically scan my codebase and understand its patterns before writing any spec
**So that** agents know how to wire code into my specific project without guessing

### Acceptance Criteria (EARS)

1. WHEN the user runs `/spec <name>` for the first time in a project that has no existing `_project-profile.md` in `.claude/specs/`
   THE SYSTEM SHALL automatically invoke the `spec-scanner` agent before proceeding to requirements gathering. The scanner SHALL read source files to answer: "How does this project register routes?", "Where do new pages get rendered?", "How does navigation work?", "Where are API endpoints defined?", "What ORM/database pattern is used?", "What entities exist and what CRUD operations are implemented for each?"

2. WHEN the user runs `/spec <name>` in a project that already has a `_project-profile.md`
   THE SYSTEM SHALL skip the full scan and proceed directly to requirements gathering, passing the existing profile to the spec-planner agent.

3. WHEN the user runs `/spec-scan` explicitly
   THE SYSTEM SHALL perform a full rescan of the codebase regardless of whether a profile already exists. The rescan SHALL merge new findings with existing manual overrides rather than overwriting the entire profile.

4. WHEN the scanner detects a monorepo with multiple application roots (identified by presence of `package.json`, `Cargo.toml`, `go.mod`, `pyproject.toml`, or similar manifest files in distinct subdirectories)
   THE SYSTEM SHALL create one profile per application root, stored as `_project-profile-<app-name>.md`.

5. WHEN the scanner encounters `.env` files, credential files, files matching `*.key`, `*.pem`, `*.secret`, or directories named `.aws`, `.gcp`, `.ssh`
   THE SYSTEM SHALL skip those files entirely and SHALL NOT include any content from them in the profile.

6. WHEN the scanner has high confidence in a pattern (3+ consistent examples found in the codebase)
   THE SYSTEM SHALL mark that pattern as `confidence: high`.

7. WHEN the scanner finds only 1-2 examples of a pattern
   THE SYSTEM SHALL mark that pattern as `confidence: medium`.

8. WHEN the scanner cannot find examples but infers a pattern from framework conventions (e.g., detecting Next.js and assuming file-based routing)
   THE SYSTEM SHALL mark that pattern as `confidence: low`.

9. IF a pattern has `confidence: low`
   WHEN the spec-planner or spec-implementer reads the profile
   THE SYSTEM SHALL surface a warning: "Low confidence pattern: [description]. Verify with user if critical to the current task." The agent MAY use AskUserQuestion to confirm.

10. WHEN the scanner runs
    THE SYSTEM SHALL complete its analysis within 2 minutes of wall-clock time, measured from the time the spec-scanner agent is invoked via the Task tool to the time `_project-profile.md` is written to disk.

## US-2: Project Profile Persistence

**As a** plugin user
**I want** the codebase knowledge to be stored in a structured, persistent file that agents read at the start of their work
**So that** every agent in every spec run has accurate, project-specific context without re-scanning

### Acceptance Criteria (EARS)

1. WHEN the scanner completes
   THE SYSTEM SHALL write the profile to `.claude/specs/_project-profile.md` containing exactly three sections: **Patterns** (human-readable descriptions of how the codebase wires things, each with a confidence level), **Entity Registry** (a markdown table with columns: Entity Name, Create, Read, Update, Delete, Notes -- where each CRUD column contains `yes`, `no`, or `partial`), and **Registration Points** (a compact list of wiring locations with `file:line` references and a one-line description of what gets registered there).

2. WHEN the profile for a single-app project exceeds 200 lines
   THE SYSTEM SHALL split it into domain-specific files: `_profile-<domain>.md` (e.g., `_profile-auth.md`, `_profile-billing.md`). Each split file SHALL contain only the patterns, entities, and registration points relevant to that domain, plus a `_profile-index.md` that lists all domain files with a one-line summary of each.

3. WHEN a user adds content under a `## Manual Overrides` section in the profile
   THE SYSTEM SHALL preserve that section verbatim during any rescan or incremental update. Manual overrides SHALL take precedence over auto-detected patterns when there is a conflict.

4. WHEN the spec-planner agent begins writing requirements
   THE SYSTEM SHALL read the project profile (or all domain profile files listed in `_profile-index.md`) and include the profile content in the planner's context.

5. WHEN the spec-implementer agent begins implementing a task
   THE SYSTEM SHALL read the project profile and use the Registration Points section to determine exactly which files and line numbers to modify when wiring new code.

6. WHEN the spec-tester agent begins verifying a task
   THE SYSTEM SHALL read the project profile's Registration Points section to check that new artifacts are registered at the expected locations.

7. THE SYSTEM SHALL NOT store any credentials, API keys, secrets, connection strings, or environment variable values in the project profile.

## US-3: Wiring-Aware Implementation

**As a** plugin user
**I want** the implementer agent to use project-specific registration points instead of a generic wiring checklist
**So that** new code is always connected to the application at the correct locations

### Acceptance Criteria (EARS)

1. WHEN the spec-implementer reads a task that creates a new artifact (route, component, endpoint, service, page, model)
   THE SYSTEM SHALL look up the matching registration point in the project profile and include the specific `file:line` location in its implementation plan before writing any code. For example, if the task creates a new page and the profile says "Routes register in `src/router.tsx:45` by adding a `<Route>` element," the implementer SHALL modify `src/router.tsx` at or near line 45.

2. WHEN the project profile contains registration points
   THE SYSTEM SHALL replace the generic wiring checklist (currently lines 40-48 in spec-implementer.md) with the project-specific registration points from the profile for the duration of that task.

3. WHEN the spec-implementer completes a task that creates a new artifact
   THE SYSTEM SHALL verify that the artifact is registered at every applicable registration point from the profile before setting `Wired: yes`. If any registration point is missed, the implementer SHALL add the registration before marking the task complete.

4. IF the project profile has no registration points (empty profile or first-ever project)
   WHEN the implementer creates a new artifact
   THE SYSTEM SHALL fall back to the existing generic wiring checklist (current behavior) and log a warning in progress.md: "No project profile available -- using generic wiring checklist."

## US-4: Verification Gates in spec-loop

**As a** plugin user
**I want** the execution loop to verify wiring after each task using project-specific registration points
**So that** wiring failures are caught immediately and fixed before proceeding to the next task

### Acceptance Criteria (EARS)

1. WHEN a task completes during spec-loop execution (after the implementer sets `Status: completed`)
   THE SYSTEM SHALL run a verification gate that checks: for each new file created or new export added by the task, is there a corresponding entry at the relevant registration points listed in the project profile? The check SHALL be incremental -- only examining artifacts created or modified by the current task, not the entire codebase.

2. WHEN the verification gate detects that a new artifact is NOT registered at an expected registration point
   THE SYSTEM SHALL block the loop from proceeding to the next task, spawn the spec-debugger agent with the specific wiring gap (artifact name, expected registration point file:line, what is missing), and wait for the debugger to fix the issue.

3. WHEN the spec-debugger fixes a wiring gap detected by a verification gate
   THE SYSTEM SHALL re-run the verification gate for that task. IF the gate passes, the loop SHALL resume with the next task. IF the gate fails again after 2 fix attempts, the system SHALL escalate by: (a) marking the task as `Status: completed` with `Wired: failed` in tasks.md, (b) logging the unresolved gap in progress.md under `## Gate Failure: <task_id>` with full details, and (c) blocking any dependent tasks from launching until the wiring gap is manually resolved.

4. WHEN the verification gate runs
   THE SYSTEM SHALL complete its check within 10 seconds of wall-clock time per task (targeted file reads, not full codebase scan).

5. IF no project profile exists
   WHEN the verification gate would normally run
   THE SYSTEM SHALL skip the gate and log: "No project profile -- verification gate skipped."

## US-5: /spec-debug Command

**As a** plugin user
**I want** a standalone command to diagnose and fix bugs within the spec context
**So that** bugs are tracked, fixed systematically, and create regression markers for future specs

### Acceptance Criteria (EARS)

1. WHEN the user runs `/spec-debug` and describes a bug (error message, unexpected behavior, or affected area)
   THE SYSTEM SHALL invoke the spec-debugger agent which: (a) investigates the bug by reading relevant source files, error logs, and stack traces; (b) identifies the root cause and affected files; (c) produces a `diagnosis.md` containing: bug ID (BUG-001, BUG-002, etc.), symptom description, root cause analysis, affected files list, and the fix strategy.

2. WHEN the debugger identifies affected files
   THE SYSTEM SHALL scan all existing `.claude/specs/*/tasks.md` files for references to those same files. IF a matching spec is found (the spec whose tasks reference the most overlapping files), the debug artifacts (`diagnosis.md`, `fix.md`) SHALL be written to that spec's directory. IF no matching spec is found, the system SHALL create a new directory `.claude/specs/debug-<slug>/` where `<slug>` is derived from the bug description (first 3-4 words, kebab-case).

3. WHEN the debugger completes its fix
   THE SYSTEM SHALL write a `fix.md` to the same directory as `diagnosis.md`, containing: the bug ID, files modified, description of each change, and a regression test description (what to check to ensure this bug does not recur).

4. WHEN the fix touches 3 or more files or requires more than one fix attempt
   THE SYSTEM SHALL automatically run `/spec-retro` for the debug session. WHEN the fix touches fewer than 3 files and succeeds on the first attempt, THE SYSTEM SHALL append a note to progress.md suggesting the user run `/spec-retro` if desired.

5. WHEN multiple bugs are reported in a single `/spec-debug` session
   THE SYSTEM SHALL assign sequential IDs (BUG-001, BUG-002, etc.) and append each diagnosis as a separate section in `diagnosis.md` rather than overwriting previous entries.

6. WHEN the user runs `/spec-debug` in a project with no existing specs (no `.claude/specs/` directory at all)
   THE SYSTEM SHALL create the `.claude/specs/debug-<slug>/` directory and operate normally in standalone mode.

7. THE SYSTEM SHALL NOT include credentials, API keys, secrets, or sensitive environment variable values in `diagnosis.md` or `fix.md`. IF a stack trace or error log contains such values, the system SHALL redact them with `[REDACTED]`.

## US-6: Dependency-Aware Parallel Execution

**As a** plugin user
**I want** spec-loop to run independent tasks in parallel
**So that** a 38-task spec completes in approximately 1-2 hours instead of 8+ hours

### Acceptance Criteria (EARS)

1. WHEN spec-loop starts an iteration
   THE SYSTEM SHALL parse the `Dependencies` field of all pending tasks in `tasks.md` to build a dependency graph. Tasks whose dependencies are all `completed` (or `none`) SHALL be identified as the "ready batch."

2. WHEN the ready batch contains more than one task
   THE SYSTEM SHALL launch up to 4 concurrent `claude --dangerously-skip-permissions` processes, each assigned a different task from the ready batch. The system SHALL NOT exceed 4 concurrent agents at any time.

3. WHEN a parallel task completes
   THE SYSTEM SHALL update `tasks.md` with that task's status. IF the completed task unblocks additional tasks, those tasks SHALL be added to the ready batch for the next scheduling cycle.

4. WHEN two or more parallel tasks modify the same file
   THE SYSTEM SHALL detect the conflict via git merge during the consolidation step. IF a merge conflict occurs, the system SHALL: (a) keep the first-committed version, (b) re-queue the conflicting task for the next iteration, and (c) log the conflict in progress.md. IF the same task is re-queued due to merge conflicts 3 times consecutively, the system SHALL log a warning "Task <ID> has conflicted 3 times -- forcing sequential execution" and run that task sequentially in the next iteration (not in a parallel batch).

5. WHEN parallel tasks complete an iteration batch
   THE SYSTEM SHALL run verification gates for each completed task before scheduling the next batch. Failed gates SHALL block dependent tasks from launching.

6. WHEN the `--no-parallel` flag is passed to spec-loop
   THE SYSTEM SHALL fall back to the current sequential one-task-per-iteration behavior.

7. WHEN parallel execution is active
   THE SYSTEM SHALL write each agent's output to a separate log file at `.claude/specs/<name>/logs/iteration-<N>-task-<T>.log` so that parallel outputs do not interleave.

## US-7: Regression Awareness

**As a** plugin user
**I want** bug fixes to create regression markers that warn future specs about fragile areas
**So that** future changes to previously-buggy files get extra scrutiny

### Acceptance Criteria (EARS)

1. WHEN `/spec-debug` completes a fix
   THE SYSTEM SHALL append a regression marker to the project profile under a `## Regression Markers` section. Each marker SHALL contain: bug ID, affected files (list), fix date, one-line description of what broke, and a regression check description (what to verify).

2. WHEN the spec-planner generates requirements for a new spec
   THE SYSTEM SHALL read the Regression Markers section of the project profile. IF any marker's affected files overlap with files likely to be modified by the new spec (based on the design), the planner SHALL include a warning in the requirements: "WARNING: [file] was involved in [BUG-ID]: [description]. Ensure [regression check] is verified."

3. WHEN the spec-tester verifies a task that modifies a file listed in a regression marker
   THE SYSTEM SHALL include the marker's regression check in its verification steps, in addition to the task's normal acceptance criteria.

4. (Deferred) Regression marker retirement (auto-suggesting removal after 90 days / 3+ clean specs) is out of scope for this spec. Markers persist until manually removed by the user.

## US-8: Gap Detection and Prerequisite Injection

**As a** plugin user
**I want** the planner to detect missing CRUD operations and other completeness gaps and add prerequisite tasks when the current feature depends on them
**So that** features work end-to-end without hitting missing infrastructure

### Acceptance Criteria (EARS)

1. WHEN the spec-planner reads the project profile's Entity Registry and the requested feature depends on an entity that has incomplete CRUD operations
   THE SYSTEM SHALL auto-add prerequisite tasks for the missing operations that the feature directly depends on. For example, if the feature needs to update invoices but the Invoice entity shows `Update: no`, the planner SHALL add a task to implement invoice updates.

2. WHEN the Entity Registry shows gaps that the current feature does NOT depend on
   THE SYSTEM SHALL include an informational note in `requirements.md` under a `## Detected Gaps (Informational)` section listing the gaps. For example: "FYI: User entity is missing Update and Delete operations. Not required for this feature."

3. THE SYSTEM SHALL NOT auto-add tasks for gaps that have no dependency relationship with the current feature's requirements.

## US-9: Automated Post-Generation Validation and Fix

**As a** plugin user
**I want** the `/spec` command to automatically validate and fix the generated spec before presenting it to me
**So that** I never have to manually run `/spec-validate` multiple times to clean up issues

### Acceptance Criteria (EARS)

1. WHEN the spec-tasker agent completes writing `tasks.md` (end of the `/spec` workflow)
   THE SYSTEM SHALL automatically invoke the spec-validator agent to check the generated requirements.md, design.md, and tasks.md for completeness, consistency, and EARS compliance.

2. WHEN the validator returns warnings or errors
   THE SYSTEM SHALL automatically invoke the spec-planner agent (for requirements/design issues) or the spec-tasker agent (for task issues) with the specific validation findings and instructions to fix them. The fix agent SHALL read the validator's report and apply targeted corrections without rewriting the entire spec.

3. WHEN the fix agent completes its corrections
   THE SYSTEM SHALL re-run the validator to confirm the issues are resolved. This validate-fix cycle SHALL repeat up to 3 times. IF issues persist after 3 cycles, the system SHALL present the remaining warnings to the user and continue.

4. WHEN the validate-fix cycle completes with zero errors and zero warnings
   THE SYSTEM SHALL skip further cycles and proceed directly to the summary.

5. WHEN the validate-fix cycle completes (either clean or after 3 max cycles)
   THE SYSTEM SHALL include the validation status in the `/spec` summary output: "Spec validated: PASS" or "Spec validated: X warnings remaining (auto-fixed Y issues)".

## Non-Functional Requirements

### NFR-1: Profile Size
THE SYSTEM SHALL keep each project profile file under 200 lines. When a file would exceed 200 lines, it SHALL be split by domain as described in US-2 AC-2.

### NFR-2: Phase 0 Performance
THE SYSTEM SHALL complete a full Phase 0 scan in under 2 minutes of wall-clock time (measured from agent invocation to profile write) for codebases up to 500 source files.

### NFR-3: Verification Gate Overhead
THE SYSTEM SHALL complete a verification gate check in under 10 seconds of wall-clock time per task.

### NFR-4: Parallel Execution Token Budget
THE SYSTEM SHALL NOT run more than 4 concurrent agent processes during parallel execution to control token cost.

### NFR-5: Idempotency
WHEN Phase 0 scan runs multiple times on the same codebase without changes
THE SYSTEM SHALL produce an identical profile each time (idempotent output).

### NFR-6: Backward Compatibility
THE SYSTEM SHALL remain fully functional for projects without a project profile. All existing commands (/spec, /spec-loop, /spec-exec, /spec-team) SHALL work as they do today when no profile exists.

### NFR-7: Security
THE SYSTEM SHALL NOT read, store, or reference credentials, secrets, API keys, or environment variable values at any point during scanning, profiling, debugging, or verification.

## Out of Scope

- Static framework templates or blueprints (everything is auto-detected via LLM heuristics)
- AST-based code parsing (agents read code as text and reason about it)
- Full end-to-end chain tracing (registration-point checks at specific file:line locations are sufficient)
- Framework-specific detector modules (no pluggable detector architecture)
- UI/dashboard for viewing profiles or debug history
- Automatic profile updates mid-implementation (verification gates do incremental checks; full rescan is manual via /spec-scan)
- Regression marker auto-retirement (suggesting removal after 90 days / 3+ clean specs -- deferred to future spec)

---

## Assumptions

1. The spec-scanner agent uses the same LLM-driven approach as other agents -- it reads code and reasons about patterns rather than using regex or AST parsing.
2. Registration points with `file:line` references are approximate -- lines may shift as code changes. Agents should search for the pattern near the indicated line rather than relying on exact line numbers.
3. Parallel execution uses separate `claude` processes (not threads within a single process), matching the existing `spec-loop.sh` pattern of invoking `claude --dangerously-skip-permissions -p`.
4. The monorepo detection heuristic (looking for manifest files in subdirectories) is sufficient to distinguish multi-app repos from single-app repos with nested dependencies.
5. The 200-line profile limit is a guideline enforced by the scanner agent, not a hard technical constraint.
