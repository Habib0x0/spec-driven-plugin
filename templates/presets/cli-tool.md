## User Stories

### US-1: Argument Parsing

**As a** CLI user
**I want** the tool to accept positional arguments and flags with sensible defaults
**So that** I can control its behavior without memorizing complex syntax

#### Acceptance Criteria (EARS Notation)

1. WHEN the tool is invoked with required positional arguments
   THE SYSTEM SHALL parse and validate each argument before executing the command

2. WHEN a required argument is missing
   THE SYSTEM SHALL print a usage message showing the expected arguments and exit with code 1

3. WHEN optional flags are omitted
   THE SYSTEM SHALL use documented default values for each flag

4. WHEN an unrecognized flag is provided
   THE SYSTEM SHALL print "Unknown flag: <flag>" followed by the usage message and exit with code 1

---

### US-2: Subcommands

**As a** CLI user
**I want** the tool to support subcommands for distinct operations
**So that** I can perform different actions with a single binary

#### Acceptance Criteria (EARS Notation)

1. WHEN the tool is invoked with a valid subcommand name
   THE SYSTEM SHALL execute the corresponding subcommand with its own argument parsing

2. WHEN the tool is invoked without a subcommand
   THE SYSTEM SHALL print the list of available subcommands with one-line descriptions and exit with code 1

3. WHEN the tool is invoked with an unrecognized subcommand
   THE SYSTEM SHALL print "Unknown command: <name>" followed by the list of available subcommands and exit with code 1

---

### US-3: Output Formatting

**As a** CLI user
**I want** to choose the output format (table, JSON, or plain text)
**So that** I can read output in the terminal or pipe it into other tools

#### Acceptance Criteria (EARS Notation)

1. WHEN the `--format table` flag is provided (or no `--format` flag, as table is the default)
   THE SYSTEM SHALL display results in an aligned table with column headers

2. WHEN the `--format json` flag is provided
   THE SYSTEM SHALL output results as valid JSON (parseable by `jq` or equivalent)

3. WHEN the `--format plain` flag is provided
   THE SYSTEM SHALL output results as tab-separated values with no headers

4. WHEN the `--format` flag is set to an unsupported value
   THE SYSTEM SHALL print "Unsupported format: <value>. Supported: table, json, plain" and exit with code 1

---

### US-4: Error Handling

**As a** CLI user
**I want** the tool to report errors clearly to stderr and exit with non-zero codes
**So that** I can detect failures in scripts and automation pipelines

#### Acceptance Criteria (EARS Notation)

1. WHEN an operation fails due to invalid input
   THE SYSTEM SHALL print the error message to stderr and exit with code 1

2. WHEN an operation fails due to a missing file or resource
   THE SYSTEM SHALL print "Error: <resource> not found" to stderr and exit with code 1

3. WHEN an unexpected internal error occurs
   THE SYSTEM SHALL print "Internal error: <message>" to stderr and exit with code 2

4. THE SYSTEM SHALL NOT print stack traces or debug information to stderr unless a `--verbose` or `--debug` flag is set

---

### US-5: Help and Version Text

**As a** CLI user
**I want** to view help text and the tool version
**So that** I can learn how to use the tool and check which version is installed

#### Acceptance Criteria (EARS Notation)

1. WHEN `--help` or `-h` is provided as a flag (globally or on any subcommand)
   THE SYSTEM SHALL print usage information including the command syntax, available flags, and descriptions, then exit with code 0

2. WHEN `--version` or `-v` is provided as a global flag
   THE SYSTEM SHALL print the tool name and version number on a single line and exit with code 0

3. WHEN `--help` is provided on a subcommand (e.g., `tool list --help`)
   THE SYSTEM SHALL print help specific to that subcommand including its flags and arguments

## Non-Functional Requirements

### Performance
- The tool MUST start and produce output within 500ms for local operations (no network calls)
- JSON output MUST be valid and parseable by standard JSON tools (jq, python json.tool)

### Compatibility
- The tool MUST run on macOS (Darwin) and Linux (x86_64, arm64)
- The tool MUST work in bash, zsh, and fish shells without requiring shell-specific configuration

### Usability
- All error messages MUST be written to stderr (not stdout) so they do not interfere with piped output
- Exit codes MUST follow convention: 0 for success, 1 for user errors, 2 for internal errors
